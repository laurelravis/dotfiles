# /기록 — 스마트 지식 기록 스킬

대화 중 발견한 설계 결정, 이슈, 정책, 기술 스펙을 Notion(+ 선택적으로 Obsidian)에
체계적으로 기록한다. 세 가지 모드로 상황에 따라 속도와 안전성을 조절한다.

---

## 모드

| 호출 | 동작 | 체감 시간 | 언제 쓰나 |
|------|------|-----------|-----------|
| `/기록` | 풀 플로우 — 계획 전문 확인 후 실행 | ~3분 | 중요한 첫 결정, 새로운 패턴, 신규 페이지/DB 필요 시 |
| `/기록 빠르게` | 한 줄 확인 후 실행, 충돌 시만 풀 플로우로 전환 | ~40초 | 일반 케이스 |
| `/기록 즉시` | 확인 없이 바로 실행, 충돌 시만 중단 | ~30초 | 소소한 반복 기록 |

> **공통 원칙**: 모드와 무관하게 템플릿 품질·미채택 옵션 기록·충돌 감지는 항상 유지한다.
> 충돌이 감지되면 어떤 모드든 자동으로 풀 플로우로 전환한다.

---

## 실행 전 환경 확인

`.claude/skill-config.json` 읽기를 먼저 시도한다.

| 상황 | 동작 |
|---|---|
| `skill-config.json` 있음 + Notion 설정 있음 | 풀 기능 — ADR DB에 기록 |
| `skill-config.json` 없음 또는 Notion 미설정 | **로컬 모드** — 프로젝트 루트 `DECISIONS.md`에 기록 |

**로컬 모드 기록 형식** (`DECISIONS.md` append):
```markdown
## [YYYY-MM-DD] [제목]

**카테고리**: [추론]
**상태**: 확정 / 검토중 / 연기됨 / 폐기됨
**목표일**: [연기 시만]

[내용]

---
```

로컬 모드에서도 연기 히스토리, 완료 표시, 상태 관리는 동일하게 동작한다. 단, 검색·집계는 수동.

---

## CONFIG

```
OBSIDIAN_VAULT_PATH = 미설정
# 설정 시 예시: /path/to/your/vault
# 미설정 상태에서는 Obsidian 단계를 건너뛴다.

OBSIDIAN_FOLDER_MAP:
  설계결정  → Decisions/
  발견·이슈 → Issues/
  운영정책  → Policies/
  기술스펙  → Specs/
```

## ADR 빠른 상태 업데이트

`/기록 [ADR 관련 처리]` 형태로 기존 결정의 상태를 즉시 바꿀 수 있다.

| 입력 예 | 동작 |
|---|---|
| "mirror 전환 한달 뒤로 미뤄" | 해당 ADR → `연기됨`, `목표일` = +30일, 본문에 연기 히스토리 한 줄 추가 |
| "Cloudflare tunnel 완료했어" | 해당 ADR → `확정`, 본문에 완료 날짜 한 줄 추가 |
| "모바일 접근 설계 안 하기로" | 해당 ADR → `폐기됨`, 본문에 폐기 사유 한 줄 추가 |

**연기 히스토리 형식** (본문 맨 위에 누적, 덮어쓰지 않음):
```
⏸ 2026-05-23 연기 — 목표일 2026-06-23 / 사유: bootstrap 안정화 우선
⏸ 2026-04-10 연기 — 목표일 2026-05-23 / 사유: 인력 부족
```

기간 계산 기준 (오늘 기준):
- "N주 뒤" → +7N일
- "한달 뒤" / "N달 뒤" → +30N일
- "다음 분기" → 다음 분기 1일
- "내년" → 내년 1월 1일

`목표일`이 지난 `연기됨` 항목은 `/파악` 실행 시 자동으로 다시 표면화된다.

## 워크스페이스 구조

`.claude/skill-config.json`에서 Notion ID를 읽는다. 프로젝트마다 아래 키를 설정해두면 된다.

| 역할 | config 키 | 언제 쓰나 |
|------|-----------|-----------|
| 결정 로그 DB | `notion.adr_db_collection` | 설계 결정 신규 항목 — 상태(검토중/확정/폐기됨) 관리 |
| 시스템 운영 가이드 | `notion.system_guide_page` | 실운영 기준 요약 섹션 추가·수정 |
| 설계 핸드오버 | `notion.handover_page` | 장기 아키텍처·인프라 결정 |
| 워크스페이스 루트 | `notion.workspace_root` | 신규 최상위 페이지 생성 시 부모 |
| 스킬 인덱스 페이지 | `notion.skill_index_page` | 스킬 추가·변경 시 목록 업데이트 |

> **신규 페이지/DB 필요 시**: 즉시 생성 금지.
> `/기록` 풀 플로우에서만 허용. 빠르게·즉시 모드에서 신규 생성이 필요하면 자동으로 풀 플로우로 전환.

## 콘텐츠 타입 분류

| 타입 | 판단 기준 | 기본 기록 위치 |
|------|-----------|----------------|
| **설계 결정** | 여러 옵션 중 하나를 선택한 기술·운영 결정 | 결정 로그 DB |
| **발견·이슈** | 버그, 예상치 못한 동작, 운영 중 발견한 문제 | 결정 로그 DB |
| **운영 정책** | 반복 적용할 규칙, 프로세스, 팀 합의 | 운영 가이드 + 결정 로그 DB |
| **기술 스펙** | 구현 방식, 파일 구조, API 설계 등 | 결정 로그 DB |

## 실행 절차

### PHASE 1 — 분석 (모든 모드 공통, 자동 실행)

1. **콘텐츠 타입 분류** — 위 기준으로 판단
2. **Notion 검색** — 관련 기존 항목 탐색 (키워드 2~3개)
3. **충돌 감지** — 기존 결정과 모순되는 내용인가?
   - 충돌 발견 시: 모드 무관하게 즉시 풀 플로우로 전환
   - "이전에 X로 결정했는데(링크), 이번 내용은 Y입니다. 기존 항목을 폐기 처리할까요?"
4. **라이프사이클 확인** — 기존 `검토중` 항목이 이제 확정되는 건가?
   - 맞다면: 신규 생성 대신 기존 항목 업데이트로 전환
5. **기록 위치 결정** — 타입별 기본 위치 적용, 신규 페이지/DB 필요 여부 판단
   - 신규 생성 필요 시: 풀 플로우로 전환

---

### PHASE 2 — 확인 (모드에 따라 다름)

#### `/기록` — 풀 확인

아래 형식으로 전체 계획을 제시하고 승인을 기다린다.

```
📋 기록 계획

타입: [설계결정 / 발견·이슈 / 운영정책 / 기술스펙]

⚠️  충돌 감지 (있을 경우만)
    이전 결정: [링크] — [내용 요약]
    → 기존 항목 폐기 처리할까요?

🔄 라이프사이클 변경 (있을 경우만)
    기존 항목 [링크] 검토중 → 확정으로 업데이트

📍 기록 위치
    1. [위치명] — [이유]
    2. [위치명] — [이유]

🆕 신규 생성 필요 항목 (있을 경우만)
    [페이지/DB 이름] — [생성 이유]
    → 생성해도 될까요?

📝 초안 내용
    [템플릿에 맞게 작성한 초안 전문]

이대로 기록할까요?
```

#### `/기록 빠르게` — 한 줄 확인

한 줄로 핵심만 요약해서 제시. 사용자가 "응" 하면 즉시 실행.
수정이 필요하면 그 자리에서 지시받아 반영 후 실행.

```
[타입] [위치] 에 기록합니다. 바로 할까요?
```

예시:
```
[설계결정] 결정 로그 DB에 기록합니다. 바로 할까요?
[운영정책] 운영 가이드 + 결정 로그 DB 두 곳에 기록합니다. 바로 할까요?
```

#### `/기록 즉시` — 확인 없이 실행

PHASE 2 없이 바로 PHASE 3으로 진행.
충돌·신규 페이지 필요 시에만 멈추고 확인.

---

### PHASE 3 — 실행 (모든 모드 공통)

1. **Notion 기록**
   - 업데이트: `notion-update-page` (라이프사이클 변경, 내용 추가)
   - 신규: `notion-create-pages` (data_source_id 지정)
2. **상호 링크 자동 삽입**
   - 복수 위치 기록 시 각 위치에 서로의 링크 삽입
3. **Obsidian 기록** (OBSIDIAN_VAULT_PATH 설정된 경우만)
   - 타입별 폴더에 `YYYY-MM-DD 제목.md` 파일 생성
   - frontmatter: `status`, `type`, `date`, `notion_url`, `tags`
   - 내부 링크: `[[페이지명]]` wikilink 스타일
4. **Claude 로컬 메모리 동기** (해당 시)
   - 이 결정이 Claude의 코딩 방식·패턴·금지 사항에 영향을 주는가?
   - 해당 시: feedback 또는 project 타입으로 메모리 저장

---

### PHASE 4 — 완료 보고 (모든 모드 공통)

```
✅ 기록 완료

[위치1]: [링크]
[위치2]: [링크]

라이프사이클: 검토중 → 확정 (변경된 경우)
메모리 동기: 저장됨 / 해당 없음
Obsidian: 저장됨 / 미설정으로 건너뜀

수정하려면: "기록 수정해줘 — [무엇을]"
```

> 빠르게·즉시 모드에서는 완료 보고에 항상 "수정하려면" 안내를 포함한다.
> 검토 기회가 줄어든 만큼, 사후 수정 진입점을 명확히 제공한다.

## 스킬 업그레이드 방법

이 파일은 `.claude/commands/기록.md` 에 있다.
대화 중 "기록 스킬에 X 기능 추가해줘" 또는 "기록 스킬 Y 부분 바꿔줘" 라고 하면
이 파일을 직접 편집해서 즉시 반영된다.

Obsidian 연동 활성화:
`OBSIDIAN_VAULT_PATH = 미설정` 을 실제 vault 경로로 교체하면 끝.
