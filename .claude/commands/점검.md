# /점검 — 코드·작업 흐름 함정 진단

주간 1회 또는 큰 작업 전 실행. `/파악`이 "지금 어디 있지?"라면 `/점검`은 "뭐가 발목 잡고 있지?"

---

## 실행 항목

### 🔴 즉시 위험 — 지금 터질 수 있는 것

**1. Notion sync 덮어쓰기 위험**
```bash
grep -E "props\[|formData\." <skill-config.json → sync.sync_file_path> | head -40
```
최근 커밋에서 위 필드들을 앱 코드에서 직접 수정했으면 🔴 경고:
`이 필드는 Notion 소유 → 다음 sync(최대 10분)에 덮어씌워짐`

**2. 고아 레코드 위험** *(테이블명은 프로젝트마다 다름 — 아래는 예시)*
```sql
SELECT COUNT(*) FROM <child_table> cs
LEFT JOIN <parent_table> cc ON cs.card_id = cc.id
WHERE cc.id IS NULL;
```

**3. 조용한 실패 구간**
```bash
# API route 중 에러 핸들링 패턴 없는 파일
grep -rL "try\|catch\|NextResponse.*error\|status.*[45][0-9][0-9]" app/api/ --include="*.ts" | head -10
```

---

### 🟡 불완전한 변경 — 나중에 조용히 깨질 것

**4. Notion property ↔ sync 매핑 드리프트**
- `skill-config.json → sync.sync_file_path` 의 `props["필드명"]` 패턴 추출
- Notion DB 실제 properties 조회 후 교차 확인
- 노션에 있는데 코드 매핑 없는 필드 → "누락된 매핑" 목록

**5. 금지 패턴 혼재 (useSession / SessionProvider)**
```bash
grep -rn "useSession\|SessionProvider" app/ components/ --include="*.tsx" --include="*.ts"
```
한 건이라도 나오면 🟡 경고 — `auth() + props` 패턴으로 교체 필요.

**6. TypeScript any 타입 무단 사용**
```bash
grep -rn ": any\|as any" app/ lib/ --include="*.ts" --include="*.tsx" | grep -v "// ok\|eslint-disable" | head -20
```

---

### 🟠 누적 마찰 — 작업 속도를 갉아먹는 것

**7. 뺑뺑이 파일** (근본 원인 미해결 신호)
```bash
git log --oneline -30 --name-only | grep "\.tsx\?\|\.ts$" | sort | uniq -c | sort -rn | head -15
```
5커밋 내 3회 이상 수정 파일 → "같은 파일을 N번 건드렸어요. 근본 원인이 있을 수 있습니다" 제안

**8. 기술 부채 마커**
```bash
grep -rn "TODO\|FIXME\|HACK\|WIP\|XXX" app/ lib/ components/ --include="*.ts" --include="*.tsx" | grep -v node_modules
```
건수 + 상위 10개 위치 출력

**9. 고아 코드** (정의됐지만 아무도 안 씀)
```bash
for f in lib/*.ts lib/*.tsx; do
  name=$(basename "$f" | sed 's/\.[^.]*$//')
  count=$(grep -rn "from.*/$name" app/ components/ lib/ --include="*.ts" --include="*.tsx" 2>/dev/null | wc -l | tr -d ' ')
  [ "$count" -eq 0 ] && echo "고아 의심: $f"
done
```

**10. 하드코딩 지뢰**
```bash
# 시간값 하드코딩 (19:00, 18:00 등)
grep -rn '"[0-9]\{2\}:[0-9]\{2\}"' app/ lib/ --include="*.ts" --include="*.tsx" | grep -v "test\|spec\|// ok" | head -15
```

**11. 같은 작업 두 가지 방식으로 처리 중인 파일 쌍**
```bash
# 비슷한 이름의 API route 중 구조가 다른 것
find app/api -name "route.ts" | xargs wc -l | sort -rn | head -20
```
크기 편차가 극단적인 파일 쌍 → 불완전한 마이그레이션 의심

**12. 리셋 버그 원인 탐지**

변경했는데 나중에 원래대로 돌아오는 현상의 주요 원인 체크:
```bash
# 같은 CSS 클래스명이 여러 파일에 정의된 것
grep -rn "className=" app/ components/ --include="*.tsx" | grep -o '"[^"]*"' | sort | uniq -d | head -10

# 같은 DB 필드를 여러 컴포넌트에서 다르게 다루는 것 (프로젝트 핵심 엔티티명으로 교체)
grep -rn "<entity_camel>\|<entity_snake>" app/ lib/ --include="*.ts" --include="*.tsx" -l
```

### 📋 관계 지도 (`/점검 관계` 로 호출 시만)

**DB 관계 맵**
```bash
grep -n "references\|\.references(" db/schema.ts
```
참조 없는 테이블(고립 데이터) 목록 출력.

**특정 파일 영향 범위** (`/점검 관계 [파일명]` 형태로 호출)
```bash
grep -rn "from.*[파일명]\|import.*[파일명]" app/ lib/ components/ --include="*.ts" --include="*.tsx"
```

### ⬜ 추후 활성화 (미구현 — 때가 되면)

- **코드-문서 정합성**: 노션 가이드 서술 vs 실제 코드 구조 비교
- **CSS 구조 복잡도**: 선택자 중복률, 파일 크기 임계치 측정
- **변경 전 자동 영향 범위 계산**: 파일 수정 전 의존 그래프 출력

📊 요약

## 실행 규칙

- 전체 목표 시간: 20~30초. grep 결과 과다 시 건수만 표시하고 상위 N개 미리보기.
- 🔴 항목은 반드시 조치 제안 한 줄 포함.
- 뺑뺑이 파일 발견 시: "근본 원인이 있을 수 있습니다 — 구조 검토가 필요한가요?" 제안.
- 이상 없으면: `✓ 점검 이상 없음 — 코드 건강 양호` 한 줄로 끝낸다.
- `/파악` 에서 코드 이상 감지 시 자동 연계. 점검 완료 후 필요시 `/파악` 재실행 제안.
