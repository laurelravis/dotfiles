# /파악 — 앱 현황 브리핑

두 가지 상황에서 쓴다:
- **세션 시작**: 전체 상태 브리핑 — 미결 결정·약속 부채·시스템 이상 한 번에 파악
- **작업 중**: "지금 잘 되고 있나?" 확인 — 방금 한 변경이 제대로 반영됐는지, 예상치 못한 부작용 없는지 빠르게 체크

두 경우 모두 Tier 1은 항상 실행. Tier 2(정합성 비교·백업)는 당일 첫 실행 시만 — 이미 실행한 당일이면 캐시 결과 표시로 빠르게 넘어간다.

---

## 프로젝트 초기 설정 자동 감지

**실행 시 가장 먼저**: `.claude/skill-config.json` 존재 여부를 확인한다.

파일이 없으면 브리핑 대신 초기화 플로우를 실행한다:

```
⚙️ 이 프로젝트에 스킬 설정이 없어요.

지금 초기화할까요? (응 / 나중에)
```

"응" 이면 아래 세 가지를 자동 수집하고 `.claude/skill-config.json`을 생성한다:

1. **프로젝트명**: `package.json`의 `name` 또는 현재 폴더명 → 자동 감지, 확인만
2. **DB URL**: `.env.local` 또는 `.env`에서 `DATABASE_URL` 자동 추출 → 없으면 입력 요청
3. **Notion ADR DB**: "이 프로젝트에 Notion ADR DB를 연결할까요?"
   - 예 + 기존 DB URL 있음 → 입력 받아 설정
   - 예 + 없음 → 지금 바로 Notion에 ADR DB 생성 후 config에 저장
   - 아니오 → notion 항목 없이 생성 (로컬 모드)

생성 후 바로 일반 브리핑 플로우를 실행한다.

"나중에" 이면 로컬 모드(git·코드 기반)로 브리핑을 진행한다.

## 수집 순서 (병렬 실행)

### Tier 1 — 항상 실행 (목표 2초)

아래 세 가지를 동시에 실행한다:

1. **git 상태**
   ```bash
   git log --oneline -8
   git status --short
   ```

2. **마지막 Notion sync 시각**
   ```sql
   SELECT value FROM system_settings WHERE key = '<sync.state_key>'
   ```
   DB URL: `skill-config.json` → `db.prod_url` (로컬 시 `db.local_url`)

3. **ADR 미결 항목**
   Notion ADR DB (`skill-config.json` → `notion.adr_db_collection`) 조회:
   - `상태 = '검토중'` 인 항목 전체
   - `상태 = '연기됨'` 이고 `목표일 ≤ 오늘` 인 항목 (기한 초과 → 다시 표면화)

### Tier 2 — 당일 1회 (system_settings `skill-config.json → sync.tier2_state_key` 키로 추적)

마지막 실행이 20시간 이상 전이면 실행. 최근 결과가 있으면 캐시 표시.

4. **Notion ↔ DB 정합성 비교**
   - Notion: `skill-config.json → sync.consultation_db_id` 로 오늘 날짜 건수 조회
   - DB: `SELECT COUNT(*) FROM <skill-config.json → sync.consultation_table> WHERE scheduled_date = CURRENT_DATE`
   - 두 값이 다르면 🔴 경고

5. **NAS 백업 상태**
   ```bash
   ssh <skill-config.json → ssh.mac_mini> "tail -3 <skill-config.json → ssh.backup_log_path>"
   ```
   마지막 백업 성공 시각 + 성공/실패 여부 파싱

### Tier 3 — 이상 감지 시만 실행

Tier 1·2에서 이상이 발견됐거나 유저가 명시적으로 요청할 때:

6. **약속 부채 스캔**
   시스템 가이드 페이지(`skill-config.json` → `notion.system_guide_page`) fetch → "예정", "추후", "계획" 키워드 추출 → ADR DB에 대응 항목 없는 것만 "미등록 약속"으로 표시

## 출력 형식

**원칙: 이상 없는 항목은 한 줄로 접는다. 이상·필요 항목만 공간을 쓴다.**

```
/파악 — YYYY-MM-DD HH:MM

✓ 시스템 이상 없음  (sync N분 전 · 노션 N건 ↔ DB N건 · 백업 [날짜])

   ── 이상 있으면 아래처럼 펼침 ──
🔴 sync 이상  마지막 실행 47분 전 · 노션 18건 ↔ DB 14건 (4건 누락 의심)
🟡 백업 경고  최근 성공 26시간 전

📦 코드
  미배포 커밋  N개  [커밋 요약 한 줄]
  uncommitted  없음 / N개 파일

⏳ 미결 결정
  · [ADR 제목]  N일째  (목표일: [날짜] | 기한 초과)

🔍 약속 부채  (미등록 N건)  ← Tier 3 실행된 경우만
  · "[키워드]"  [출처 섹션]

💡 지금 결정하면 좋은 것
  1. [항목]
  2. [항목]
```

## 인터랙션 처리

브리핑 출력 후 유저가 미결 결정이나 약속 부채에 대해 응답하면 즉시 처리한다.

| 유저 응답 | 처리 |
|---|---|
| "한달 뒤", "N주 뒤", "다음 분기" | ADR `상태` → `연기됨`, `목표일` 계산, 본문에 연기 히스토리 한 줄 추가 |
| "했어", "완료", "됐어" | ADR `상태` → `확정`, 본문에 완료 날짜 추가 |
| "안 해", "폐기", "취소" | ADR `상태` → `폐기됨`, 사유 기록 |
| "ADR 등록" (약속 부채) | `/기록` 플로우로 연결 |
| "무시" / "나중에" | 해당 항목 이번 세션에서 숨김 (DB 변경 없음) |

처리 완료 후 확인 한 줄:
```
✅ "[ADR 제목]" → 연기됨 (목표일 2026-06-23)
```

## 주의사항

- DB 쿼리는 `skill-config.json → db.prod_url` / `db.local_url` 참조. 접속 포트는 URL에서 읽는다.
- Notion API 오류 시 해당 항목만 "조회 실패" 표시하고 나머지 진행.
- 출력은 30줄 이내로 유지. 길어지면 항목 접어서 요약.
- `/점검` 이 필요한 상황(코드 건강 이상 감지)이면 마지막 줄에 제안: `코드 이상 감지 → /점검 실행 권장`
