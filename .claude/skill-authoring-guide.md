# 스킬 작성 가이드

`~/.claude/commands/*.md` 스킬 파일이 어떻게 설계되었는지 설명한다.  
다른 프로젝트에 붙이거나 새 스킬을 만들 때 참고한다.

---

## 스킬이란

Claude Code의 슬래시 커맨드(`/기록`, `/파악`, `/점검`)를 정의하는 마크다운 파일.  
파일 내용이 곧 Claude에게 주는 실행 지침이다.  
`~/.claude/commands/` 에 두면 모든 프로젝트에서 사용 가능.  
프로젝트 전용으로 만들려면 프로젝트 루트 `.claude/commands/` 에 두면 된다.

---

## 파일 구조 (공통 골격)

```
# /스킬명 — 한 줄 설명

[스킬 소개 — 언제 쓰는지, 두 줄 이내]

---

## 모드 (선택)
[여러 호출 방식이 있는 경우 표로 정리]

---

## CONFIG
[이 스킬이 읽는 설정값 목록 — 실제 값은 절대 쓰지 않음]

---

## 실행 항목 / 수집 순서
[단계별 실행 내용 — PHASE 또는 Tier 방식]

---

## 출력 형식
[사용자에게 보여줄 결과 템플릿]

---

## 실행 규칙
[예외 처리, 시간 목표, 다른 스킬과의 연계]
```

---

## 세 스킬의 역할 분담

```
/파악  →  "지금 어디 있지?"  지금 상태 브리핑 (git, sync, ADR 미결)
/점검  →  "뭐가 발목 잡지?"  코드 건강 진단 (위험·부채·마찰)
/기록  →  "이걸 남겨둬야지"  설계 결정·이슈·정책을 Notion에 기록
```

세 스킬은 서로 연계된다:
- `/파악` 이 코드 이상 감지 → `/점검` 실행 제안
- `/점검` 이 구조 문제 발견 → `/기록` 으로 ADR 등록 제안
- `/파악` 의 미결 결정 → 응답하면 자동으로 `/기록` PHASE 3 실행

---

## /기록 설계 포인트

### 세 가지 속도 모드

| 호출 | 확인 방식 | 언제 |
|------|-----------|------|
| `/기록` | 전체 계획 미리보기 후 승인 | 중요한 결정, 신규 페이지 필요 시 |
| `/기록 빠르게` | 한 줄 요약 확인 | 일반 케이스 |
| `/기록 즉시` | 확인 없이 바로 실행 | 반복 기록 |

충돌·신규 페이지 생성 필요 시 어떤 모드든 자동으로 풀 플로우로 격상.

### PHASE 구조

```
PHASE 1  분석    — 타입 분류, 기존 항목 검색, 충돌 감지, 라이프사이클 확인
PHASE 2  확인    — 모드에 따라 확인 수준 결정
PHASE 3  실행    — Notion 기록, 상호 링크, Obsidian 연동, Claude 메모리 동기
PHASE 4  완료 보고 — 링크, 라이프사이클 변경 요약
```

### 콘텐츠 타입 4종

| 타입 | 특징 |
|------|------|
| 설계 결정 | 미채택 옵션도 반드시 포함 |
| 발견·이슈 | 영향 범위 필수 |
| 운영 정책 | 예외 사항 명시 |
| 기술 스펙 | 롤백 방법 포함 |

## /파악 설계 포인트

### Tier 구조 (비용 최적화)

```
Tier 1  항상 실행   — git 상태, 마지막 sync 시각, ADR 미결 항목 (목표 2초)
Tier 2  당일 1회    — Notion ↔ DB 정합성, 백업 상태 (캐시 키로 중복 방지)
Tier 3  이상 시만   — 약속 부채 스캔 (비용 높음)
```

Tier 2 캐시: `system_settings` 테이블에 마지막 실행 시각 저장. 20시간 이내면 재실행 건너뜀.

### 출력 원칙

이상 없는 항목은 한 줄로 압축. 이상·액션 필요 항목만 공간 사용.  
30줄 이내 유지.

### config 의존성

모든 접속 정보는 `skill-config.json`에서만 읽음. 파일 없으면 초기화 플로우 실행.

---

## /점검 설계 포인트

### 위험 등급 3단계

```
🔴 즉시 위험   — Notion sync 덮어쓰기, 고아 레코드, 조용한 실패 (지금 터질 수 있음)
🟡 불완전한 변경 — 매핑 누락, 금지 패턴, any 타입 (나중에 조용히 깨질 것)
🟠 누적 마찰   — 뺑뺑이 파일, TODO, 고아 코드, 하드코딩 (작업 속도를 갉아먹음)
```

### 프로젝트 특정 항목 표시

프로젝트마다 다른 테이블명·파일명은 `skill-config.json`에서 읽거나  
실행 시 플레이스홀더(`<child_table>`, `<entity_snake>`)를 교체해서 사용.

---

## skill-config.json 패턴

스킬 파일에 직접 값을 쓰지 않고 config를 경유하는 이유:
- 스킬 파일은 dotfiles에 공개 커밋 가능 → 보안 정보 없어야 함
- config는 프로젝트 로컬에만 존재 → gitignore 처리

```
스킬 파일(공개)  →  읽기  →  skill-config.json(로컬 비공개)  →  읽기  →  실제 값
```

config 파일 위치: 프로젝트 루트 `.claude/skill-config.json`  
템플릿 위치: `.claude/skill-config.example.json` (커밋 허용)

---

## 새 스킬 만드는 법

1. `~/.claude/commands/<스킬명>.md` 파일 생성
2. 위 골격 복사 후 채우기
3. 민감 값은 전부 `skill-config.json → <key>` 형태로 참조
4. `skill-config.example.json`에 새 키 플레이스홀더 추가
5. `skill-security-checklist.md` 체크리스트 실행

스킬 내용 변경: 대화 중 "기록 스킬에 X 추가해줘" 하면 Claude가 직접 파일 편집.
